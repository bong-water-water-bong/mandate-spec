# Mandate — v0 Spec

**Date:** 2026-05-03
**Status:** Draft, pending review
**Owner:** bong-water-water-bong / 1bit.systems
**Consumers:** 1bit.vault (preferences + archive), 1bit-pay (settlement), 1bit.systems (agent runtime), arbitrary third-party merchants

---

## 1. Motivation

The agent-mediated commerce stack the rest of the industry is building (Stripe Link Wallet, Visa Trusted Agent, Mastercard Agent Pay, Google ACP, OpenAI's Instant Checkout) all sit on the **merchant** side of the wire. Each one is asking "how do we accept agent traffic safely?" None of them is shipping the **buyer-side** primitive: the portable, signed, attested, budgeted artifact a buyer's agent carries into any merchant on the buyer's behalf.

That artifact is a *mandate*. It is what makes agent-side intent independently verifiable, transportable across merchants, and bindable to a settlement rail without ceding custody of the buyer's wallet to any single payment network.

A mandate is not an order, not a quote, not a payment authorization, and not a chat session. It is a signed brief from the buyer to *some* future merchant that says: *here is who I am, here is what I want, here is how much I will spend, here is the agent runtime that minted this on my behalf, and here is the proof you can use to trust all of the above.*

`1bit-pay` is the settlement spine. The mandate is the wire format that flows on top of it. Together they let 1bit.systems' agent runtime transact with merchants who have never heard of us, using merchants' own attested receipts to close the audit loop back into 1bit.vault.

## 2. Scope

**In scope (v0):**
- Mandate JSON schema, canonical serialization, content-addressed identifier
- Buyer wallet signature binding (CAIP-10 chain-agnostic addresses; SIWE/SIWS-style sig)
- Agent runtime attestation binding (SEV-SNP report_data → mandate hash)
- Vault preference pointer + scoped read grant (UCAN-style capability)
- Mandate lifecycle (mint → present → quote → approve → fulfill → settle → receipt)
- Reference HTTP transport (`.well-known/mandate/v0/*`) + merchant capability discovery
- Settlement integration with 1bit-pay (new ledger entry kind, mandate-bound payment intent)
- Merchant signed receipt format that closes back to mandate.id and 1bit-pay ledger entry

**Explicitly out of scope (v0):**
- Multi-merchant comparison flow (mint once, broadcast to N, accept one) — v1
- Recurring / subscription mandates — v1
- Outcome-based mandates (pay-on-eval, pay-on-delivery-confirmed) — v1
- Sub-mandates / delegated budgets (parent agent → child agent) — v1
- Mandate marketplaces / discovery directories — v2
- ACP envelope encoding — v0.2 once Google's ACP surface stabilises
- Privacy-preserving mandates (ZK intent without revealing brief contents) — research track

**Non-goals:**
- Don't replicate 1bit-pay. Mandate is a wire format + lifecycle; 1bit-pay is the settlement substrate. Settlement details live in `2026-05-03-1bit-pay-design.md`.
- Don't compete with ACP or x402 as a transport. Mandate is a payload; it rides over either.
- Don't try to reach pre-existing merchants without an SDK. v0 lands one real merchant pilot; broader merchant onboarding is a v0.3 question.

## 3. Architecture

```
              buyer (wallet sig)
                     │
                     ▼
        ┌────────────────────────┐
        │   mandate-agent        │
        │   (1bit.systems        │
        │    runtime, SEV-SNP)   │
        └──────┬─────────────────┘
               │ reads prefs
               ▼
        ┌────────────────────────┐                 ┌──────────────────┐
        │   1bit.vault           │                 │  merchant        │
        │   (prefs + grants +    │                 │  /.well-known/   │
        │   mandate archive)     │                 │   mandate/v0/*   │
        └────────────────────────┘                 └─────┬────────────┘
                                                        │
        mandate://b3:<hash> ────[present]──────────────▶│
                                                        │ quote
                              ◀────────[quote sig]──────┤
                                                        │
                  [approve sig (if over threshold)]────▶│
                                                        │
                              ◀──────[fulfill intent]───┤
                                                        │
                                                        ▼
                                            ┌───────────────────────┐
                                            │   1bit-pay            │
                                            │   ledger + rail       │
                                            │   (x402/USDC/Stripe)  │
                                            └───┬───────────────────┘
                                                │ settle
                              ◀────[merchant signed receipt]────────┤
                                                │
                                                ▼
                                       writes to 1bit.vault
                                       receipts/<wallet>/<mandate-id>
```

**Layering:**

1. **Identity** — buyer wallet (CAIP-10) signs the mandate body. Wallet sig is the root of buyer authority.
2. **Brief** — natural-language intent + structured constraints, sourced from 1bit.vault preferences.
3. **Envelope** — budget cap, settlement rail prefs, fulfillment policies, expiry, nonce.
4. **Witness** — agent runtime attestation (SEV-SNP report quote with `report_data` bound to mandate hash) certifies that the brief was minted by a known runtime, not a person hand-crafting a fake.
5. **Transport** — HTTP to merchant well-known endpoints; merchant returns signed quote → buyer or agent approves → fulfillment + settlement → merchant signed receipt.
6. **Settlement** — 1bit-pay creates a mandate-bound payment intent; rail (x402, Stripe, USDC) executes; receipt closes the loop.
7. **Archive** — mandate + receipt + preference snapshot land in 1bit.vault as a tamper-evident audit object.

## 4. The mandate

### 4.1 Identifier

`mandate://b3:<base32-blake3-hash>` — content-addressed against the canonical serialization of the mandate body (excluding signatures, since signatures cover the body).

Multiple representations are explicitly allowed:
- `mandate://b3:...` — direct reference
- `ipfs://<cid>` — when archived to IPFS post-close
- DID document fragment — for runtime addressing

### 4.2 Canonical serialization

JCS (RFC 8785) over JSON. All signatures are computed over the JCS-canonical bytes of the body with the relevant signature fields set to `null`.

### 4.3 Schema

```json
{

  "id": "mandate://b3:abc123...",

  "buyer": {

      "alg": "eip191",

  "intent": {

    }

  "approval": {
    "human_required_above": { "amount": "2000", "currency": "USD-cents" },
    "approval_sig": null
  },

  "lifecycle": {
    "created_at": "2026-05-03T16:00:00Z",
    "expires_at": "2026-05-10T16:00:00Z",
    "nonce": "0x...",
    "bound_to_merchant": null,
    "state": "minted"
  }
}
```

### 4.4 Field semantics

| Field | Meaning | Mutability |
|---|---|---|
| `id` | Content hash of the canonical body. Recomputable; merchants verify on receipt. | Immutable post-mint |
| `buyer.wallet` | CAIP-10 identifier. Chain-agnostic. | Immutable |
| `buyer.sig` | Buyer's wallet signature over the canonical body with all `*.sig` and `lifecycle.bound_to_merchant` set to `null`, and `approval.approval_sig` set to `null`. | Immutable |
| `intent.brief` | Free-text intent. Merchant-readable, agent-summarized. | Immutable |
| `intent.structured` | `must` (hard constraints), `prefer` (soft), `avoid` (negative). Merchants reject if `must` cannot be satisfied. | Immutable |
| `budget.max_total` | Hard cap, fees inclusive (when `fees_inclusive=true`). Merchant cannot quote above. | Immutable |
| `budget.settlement_rails` | Ordered preference list. Merchant picks the highest-priority rail it accepts. | Immutable |
| `preferences.vault_pointer` | CID to a preferences document the merchant *may* read for ranking. | Immutable |
| `preferences.vault_grant` | UCAN bounded to (mandate.id, merchant DID, expiry). Merchant fetches with this; vault enforces scope. | Immutable |
| `policies.substitutions` | `ok` \| `no` \| `ask`. `ask` forces an approval round-trip. | Immutable |
| `agent.attestation.report_data_binding` | First 64 bytes of `blake3(canonical_body)` MUST equal the SEV-SNP report's `REPORT_DATA` field. Verifies the runtime minted *this exact* mandate. | Immutable |
| `approval.human_required_above` | If quote total exceeds this, merchant MUST require `approval.approval_sig` before fulfilling. | Immutable |
| `approval.approval_sig` | Buyer wallet signature over `(mandate.id, quote.id, quote.total)`. Set during the approve step. | Mutable (set once) |
| `lifecycle.bound_to_merchant` | Set on first accepted quote. After binding, mandate is no longer presentable to other merchants. | Mutable (set once) |
| `lifecycle.state` | One of: `minted`, `presented`, `quoted`, `approved`, `fulfilling`, `settled`, `closed`, `expired`, `rejected`, `refunded`. | Mutable |

### 4.5 Signing rules

- **Buyer body signature** — algorithm declared in `buyer.sig.alg`. v0 supports `eip191` (EVM wallets, SIWE-style) and `ed25519-solana` (SIWS). Sign over JCS-canonical body with `buyer.sig.value`, `approval.approval_sig`, and `lifecycle.bound_to_merchant` all set to `null`.
- **Approval signature** — sign over `blake3(mandate.id || quote.id || quote.total_canonical_bytes)` with the same wallet. Algorithm matches `buyer.sig.alg`.
- **Attestation binding** — runtime computes `blake3(canonical_body)`, takes the first 64 bytes, and submits that as `REPORT_DATA` when requesting the SEV-SNP attestation report. Verifier recomputes and checks.
- **Merchant receipt** — Ed25519 over a defined receipt body (§4.7).

### 4.7 Quote, fulfillment, and receipt bodies

**Quote** (merchant → agent, signed):
```json
{
  "quote_id": "uuid",
  "mandate_id": "mandate://b3:...",
  "merchant_did": "did:web:roaster.example.com",
  "line_items": [...],
  "total": { "amount": "3850", "currency": "USD-cents" },
  "fees_breakdown": [...],
  "fulfillment_eta": "2026-05-08T...Z",
  "rail": "x402+usdc-base",
  "settle_endpoint": "...",
  "expires_at": "2026-05-03T16:30:00Z",
  "merchant_sig": "ed25519:..."
}
```

**Fulfillment intent** (merchant → agent, signed, post-approval):
```json
{
  "fulfillment_id": "uuid",
  "mandate_id": "mandate://b3:...",
  "quote_id": "uuid",
  "rail": "x402+usdc-base",
  "settle_request": { /* rail-specific payload */ },
  "merchant_sig": "ed25519:..."
}
```

**Receipt** (merchant → agent, signed, post-settlement):
```json
{
  "receipt_id": "uuid",
  "mandate_id": "mandate://b3:...",
  "quote_id": "uuid",
  "fulfillment_id": "uuid",
  "settled_amount": { "amount": "3850", "currency": "USD-cents" },
  "settled_rail": "x402+usdc-base",
  "settled_tx": "0x...",
  "1bit_pay_ledger_entry_id": "uuid",
  "fulfilled_at": "2026-05-03T16:45:00Z",
  "merchant_sig": "ed25519:...",
  "merchant_did": "did:web:roaster.example.com"
}
```

The receipt is the bidirectional close: merchant attests fulfillment + settlement; 1bit-pay's signed inference-/payment-receipt attests the buyer's side. Both reference `mandate.id`.

## 5. Lifecycle

```
   mint ──▶ present ──▶ quote ──┬──▶ approve ──▶ fulfill ──▶ settle ──▶ closed
                                │
                                └──▶ rejected
                       (any state) ──▶ expired   (after lifecycle.expires_at)
                       (settled)   ──▶ refunded  (merchant-initiated)
```

1. **mint** — agent reads vault prefs, drafts brief, attaches budget+policies, gets buyer wallet sig, attaches SEV-SNP attestation. State: `minted`.
2. **present** — agent POSTs mandate to merchant `/quote`. State: `presented`.
3. **quote** — merchant returns signed quote bound to mandate.id. Sets `lifecycle.bound_to_merchant` on agent-side mandate copy. State: `quoted`.
4. **approve** — if `quote.total > approval.human_required_above`, agent surfaces to user; user wallet-signs; agent POSTs `approval_sig` to `/approve`. State: `approved`. (If under threshold, skip — mandate's existing wallet sig already authorises within budget.)
5. **fulfill** — merchant returns fulfillment intent with rail-specific settle payload. State: `fulfilling`.
6. **settle** — agent invokes 1bit-pay to settle on the chosen rail; 1bit-pay returns ledger entry id + on-chain tx (if applicable); agent POSTs settlement proof to merchant `/settle`. State: `settled`.
7. **close** — merchant returns signed receipt; agent archives mandate + quote + fulfillment + receipt to 1bit.vault under `mandates/<wallet>/<mandate-id>/`. State: `closed`.

Idempotency: every step is keyed by `mandate.id` plus the step name. Replays return the previously committed result.

## 6. Integration with 1bit-pay

- New ledger entry kind: `mandate_settle`. Payload: `{ mandate_id, merchant_did, quote_id, rail, amount }`.
- 1bit-pay exposes `POST /v0/mandate/settle` taking `(mandate, fulfillment_intent)`. Returns `{ ledger_entry_id, rail_tx, signed_pay_receipt }`.
- The 1bit-pay signed receipt (per §4.5 of the 1bit-pay design doc) gets a new optional field: `mandate_id`. When present, the receipt is jointly indexable in vault under both `receipts/` (1bit-pay side) and `mandates/` (mandate side).
- Spend governance composes: a mandate cannot settle through 1bit-pay above the buyer's per-key cap, regardless of the mandate's declared `budget.max_total`. The tighter of the two wins.

## 7. Integration with 1bit.vault

- `preferences.vault_pointer` is a CID into the buyer's vault preferences namespace. Recommended layout: `prefs/<category>/profile`.
- `preferences.vault_grant` is a UCAN scoped to `(mandate.id, merchant.did, expires_at)`. Vault enforces the scope server-side; merchant cannot widen the read with a leaked grant.
- Closed mandates are written back to vault under `mandates/<wallet>/<mandate-id>/` containing: the mandate body, the quote, the approval (if any), the fulfillment intent, the merchant receipt, and the 1bit-pay receipt. This is the buyer's complete audit object.
- Vault's LLM-compression pipeline can summarize closed mandates into the buyer's preference document, so future mandates inherit learned reputation ("merchant X delivered on time, on spec, no substitutions"). This is how brand-becomes-buyer-memory operationally happens.

## 8. Integration with 1bit.systems / runtime

- `mandate-agent` is a new component on the 1bit runtime. Runs inside the same SEV-SNP CVM as inference workloads.
- Agent uses 1bit-runtime inference to draft the structured `intent` from the buyer's free-text request + vault prefs.
- Attestation report is produced by the same SEV-SNP context that signs inference receipts. Single attestation root for inference and mandates.
- Agent keys are short-lived per-runtime-boot ed25519, attested at boot. Mandate signatures use the buyer wallet — the agent does not sign mandate bodies, only attests that *it* minted *this* body.

## 9. Threat model

| Threat | Defense |
|---|---|
| Merchant inflates quote above mandate budget | `budget.max_total` is signed; merchant signature on quote with `total > max_total` is self-incriminating. Agent rejects. |
| Merchant substitutes without permission | `policies.substitutions` is signed. `no` blocks; `ask` forces an approve round-trip. Receipt mismatch is detectable post-fact. |
| Merchant replays mandate against itself | Idempotency by `mandate.id` + `step`; merchant cannot extract a second settlement. |
| Mandate replayed across merchants | `lifecycle.bound_to_merchant` set on first accepted quote; any other merchant rejects. Buyer agent enforces single-bind locally. |
| Forged buyer sig (no wallet) | Standard wallet sig recovery on `buyer.sig`; no recovery, no accept. |
| Forged agent attestation | SEV-SNP VCEK chain validated against AMD root; `report_data` recomputed and checked against canonical body hash. |
| Compromised agent runtime exfils prefs | Vault grant is UCAN-bounded to `(mandate.id, merchant.did)`; vault rejects out-of-scope reads. Compromise still leaks one mandate's prefs but not the buyer's history. |
| Buyer disavows after settlement | Wallet sig is non-repudiable on the agreed body; merchant retains receipt + signed quote + (if applicable) approval sig. |
| Front-running a mandate | Mandates are presented privately point-to-point; not broadcast in v0. Multi-present is a v1 design problem with disclosure controls. |
| Merchant fakes receipt to claim non-fulfillment | 1bit-pay rail-side proof (on-chain tx hash for x402/USDC, charge id for Stripe) is independent of merchant signature. Receipt + rail proof must agree. |
| Merchant signing key compromise | Merchant DID document SHOULD support key rotation; mandate verification pins key version at quote time. |
| Buyer wallet theft | Out of scope for mandate; same threat as any wallet-authed system. Mitigated by the buyer's wallet security, not the mandate spec. |

## 10. Phasing

**v0.0 — spec + libs (1–2 weeks)**
- This document.
- `mandate-spec` repo: schema (JSON Schema), canonical serializer (Rust + TS), Blake3-based id, JCS dependency.
- `mandate-core` Rust crate: types, sign/verify, attestation parse + verify (reuse `1bit-pay-attest`).
- TypeScript port for browser + Node.

**v0.1 — agent runtime reference (2–3 weeks)**
- `mandate-agent` binary on the 1bit runtime: vault read, intent drafting via local model, mandate mint, attestation capture, HTTP present/approve/fulfill/settle.
- Integration with `1bit-pay-x402` for settlement.
- Local CLI for human-driven mandate inspection / approval.

**v0.2 — merchant SDK (2 weeks)**
- `mandate-merchant-ts` and `mandate-merchant-py`: well-known endpoint scaffolds, signed quote/fulfillment/receipt helpers, attestation verifier.
- Reference merchant: a fake roaster that returns deterministic quotes + fake fulfillment.

**v0.3 — pilot (variable)**
- One real merchant pilot. Likely candidate: a specialty roaster relationship the buyer already has. Build their integration as the launch case study.
- End-to-end: `mandate-agent` on Strix Halo box reads vault coffee prefs, mints mandate, presents to roaster, settles via x402+USDC on Base, archives in vault. Demoable.

**v1 — multi-merchant + recurring + outcome**
- Multi-present with disclosure controls.
- Recurring mandates (`lifecycle.repeat_policy`).
- Outcome-based settlement (eval-witness signature gates settle).
- ACP envelope encoding.

## 11. Open questions

- **Merchant identity rooting**: `did:web` is fine for v0 but not great long-term. Should the merchant DID be anchored on-chain (cheap), in a registry, or left open?
- **Pricing for mandate-merchant SDK**: free + take-rate via 1bit-pay is the obvious model, but what's the right take rate when the merchant chose a non-1bit-pay rail (e.g. their own Stripe)? Default to 0% in that case and monetize only the 1bit-pay-rail path.
- **Categories taxonomy**: who curates `intent.structured.category`? v0 ships a starter set; longer term this is a directory problem. Avoid building a directory in v0.
- **Approval UX**: agent-side approval is wallet-sign over `(mandate.id, quote.id, quote.total)`. What's the minimum UI to make this not painful? Probably a 1bit.vault notification surface.
- **Quote-to-approval window**: quotes have an `expires_at`. Tight (1–2 min) is good for fast-moving inventory; loose (hours) is good for human approval cycles. Default 5 minutes; let merchant override.
- **Vault grant on quote-rejection**: if the merchant fetched prefs to compute a quote and then the quote is rejected, the merchant has read the prefs anyway. Live with this in v0; mark as known leak.

## 12. Decisions made

- **Repo:** `bong-water-water-bong/mandate-spec` (public, MIT, schema + libs). Pilot impls live in `bong-water-water-bong/mandate-agent` and `bong-water-water-bong/mandate-merchant`.
- **Why public:** the wedge requires merchants to adopt this. Closed spec = no adoption. The monetizable surface is settlement (1bit-pay) and agent runtime (1bit.systems), not the wire format.
- **Hash:** Blake3.
- **Canonical JSON:** JCS (RFC 8785).
- **Buyer sig:** EIP-191 (EVM) + Ed25519 (Solana) in v0. Add CAIP-122 once mature.
- **Agent attestation:** AMD SEV-SNP via the same pipeline as 1bit-pay's signed inference receipts. Reuse, don't duplicate.
- **Vault grant:** UCAN v0.10.
- **Transport:** HTTP + `.well-known/mandate.json`. ACP and x402-as-transport deferred.
- **Settlement integration:** new `mandate_settle` ledger entry kind in 1bit-pay; `mandate_id` becomes an optional field on the existing 1bit-pay signed receipt.
